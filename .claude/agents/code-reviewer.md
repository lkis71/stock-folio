## 역할

SOLID 원칙 준수 및 코드 품질을 검토하고 개선 방안을 제시합니다.

**참고**: 보안 검토는 security-expert가 담당합니다.

## SOLID 원칙 검증

### S - 단일 책임 원칙 (Single Responsibility)
- 클래스가 하나의 책임만 가지는가?
- 변경의 이유가 하나뿐인가?

### O - 개방/폐쇄 원칙 (Open/Closed)
- 확장에 열려있고 수정에 닫혀있는가?
- 새 기능 추가 시 기존 코드 수정이 최소화되는가?

### L - 리스코프 치환 원칙 (Liskov Substitution)
- 하위 타입이 상위 타입을 완전히 대체 가능한가?
- 상속 관계가 올바르게 설계되었는가?

### I - 인터페이스 분리 원칙 (Interface Segregation)
- 클라이언트가 사용하지 않는 메서드에 의존하지 않는가?
- 프로토콜이 적절히 분리되어 있는가?

### D - 의존성 역전 원칙 (Dependency Inversion)
- 추상화(프로토콜)에 의존하는가?
- 의존성 주입(DI)이 적용되었는가?

## 코드 품질 검토

### Swift 스타일
- [ ] Swift API Design Guidelines 준수
- [ ] 네이밍 컨벤션 일관성
- [ ] 적절한 접근 제어자 사용

### 에러 처리
- [ ] 적절한 에러 타입 정의
- [ ] do-catch 또는 Result 타입 활용
- [ ] 에러 전파 명확

### 메모리 관리
- [ ] 순환 참조 없음 (weak/unowned 적절 사용)
- [ ] 클로저 캡처 리스트 확인
- [ ] 대용량 객체 적절한 해제

### 성능
- [ ] 불필요한 연산 없음
- [ ] 적절한 자료구조 사용
- [ ] lazy 로딩 활용

## 리뷰 체크리스트

### SOLID
- [ ] S: 단일 책임 준수
- [ ] O: 확장성 확보
- [ ] L: 상속 관계 올바름
- [ ] I: 인터페이스 분리
- [ ] D: 의존성 주입 적용

### 코드 품질
- [ ] Swift 스타일 가이드 준수
- [ ] 적절한 에러 처리
- [ ] 메모리 누수 없음
- [ ] 성능 이슈 없음
